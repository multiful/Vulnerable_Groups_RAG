## 5. 기능 상세

## F-01. 위험군 기반 추천 진입 (설문 방식)

### 목적
12문항 설문을 통해 사용자의 위험군 단계(1~5)를 판정하고 추천 흐름을 시작한다.

### 주 사용자
- 사용자

### 입력
12문항 설문 응답 (각 문항은 카테고리별 점수 옵션으로 구성)

| 카테고리 | 문항 수 | 주요 측정 내용 |
|---|:---:|---|
| 관계망 | 4 | 조언·도움·감정 공유 가능한 사람 유무, 지인 만남 빈도 |
| 활동 | 2 | 외출 빈도, 대화 빈도 |
| 노동·경제 | 2 | 취업 상태, 경제 상황 인식 |
| 정신건강 | 3 | 우울감, 흥미 저하, 자살·자해 사고 (Safety Key) |
| 자기관리 | 1 | 식사·위생·정리 등 일상 관리 수준 |

### 출력
- 판정된 위험군 단계 (1~5)
- 카테고리별 점수 요약
- Safety Override 활성 여부

### 처리 규칙
- 위험군은 1단계 ~ 5단계 구조를 따른다.
- 단계 판정: `총점 / 최대점수` 비율로 5개 구간으로 분류한다.
  - 0%~19%: 1단계, 20%~37%: 2단계, 38%~55%: 3단계, 56%~73%: 4단계, 74%~100%: 5단계
- **Safety Override**: 정신건강 항목 중 `B12_9`(자살·자해 사고)에서 score ≥ 2 ("일주일 이상") 이상 응답 시 Safety Flag를 활성화한다. Safety Flag 활성 시 비율 점수가 80% 미만이어도 최소 4단계(60% 미만)~5단계(60% 이상)로 조정한다.
- Safety Flag 활성 시 위기 상담 기관 안내 배너(1393 자살예방상담전화)를 함께 표시한다.
- 판정된 위험군 단계는 이후 F-02(도메인·직무 선택) 진입 시 URL 파라미터로 전달한다.

### 예외 상황
- 미응답 문항이 있는 상태에서 결과 보기 시도 → 미응답 문항으로 이동
- 허용 범위를 벗어난 단계값
- 전체 재진단 요청 (다시 진단 버튼)

### 현재 상태
활성

## F-02. 관심 도메인·직무 선택 (InterestSelection)

### 목적
F-01에서 판정된 위험군 단계를 바탕으로, 사용자가 관심 도메인과 직무를 선택하여 추천 맥락을 완성한다.

### 주 사용자
- 사용자

### 입력
- F-01에서 전달된 위험군 단계 (URL 파라미터 `stage`)
- 관심 도메인 선택 (필수 — domain taxonomy 세부 라벨 중 1개)
- 관심 직무 선택 (선택 — job taxonomy 세부 직무 중 1개 이상)

### 출력
- 선택된 도메인 ID (`domain_XXXX`)
- 선택된 직무 ID 목록 (`job_XXXX`) (선택 입력)
- F-05(로드맵 조회)로 전달할 맥락: `stage` + `domain` + `domainName` (+ `job`이 있으면 추가)

### 처리 규칙
- `related_domains`는 허용된 domain taxonomy 세부 라벨만 사용한다. 자유 텍스트 도메인 생성 금지.
- `related_jobs`는 허용된 희망 직무 taxonomy 세부 직무만 사용한다. 자유 텍스트 직무 생성 금지.
- 도메인은 단일 선택 필수, 직무는 선택적으로 추가한다.
- 도메인 선택 없이 다음 단계로 이동할 수 없다.
- 도메인만 선택하고 직무를 선택하지 않아도 로드맵 진입 가능하다 (위험군 × 도메인 기반 추천).
- 선택된 값은 URL 파라미터로 Roadmap으로 전달한다 (store 미사용, URL 기반 상태 전달).

### 예외 상황
- 도메인 미선택 상태에서 다음 이동 시도 → 버튼 비활성화
- 직무/도메인 값이 taxonomy 밖 값인 경우 → 오류로 간주, 선택 불가 처리
- 위험군 단계 파라미터 누락 시 → 단계 없이 도메인·직무 기준만으로 추천 진입

### 현재 상태
활성 (도메인 선택 구현됨 / 직무 선택 UI는 추가 구현 필요)

## F-03. 자격증 후보 결과 조회

### 목적
정규화된 추천 맥락을 바탕으로 자격증 후보를 반환한다.

### 주 사용자
- 사용자

### 입력
- 위험군 단계
- 정규화된 관심 직무
- 정규화된 관심 도메인
- 추천 요청 맥락

### 출력
추천 후보 리스트. 각 후보는 최소 아래 정보를 포함한다.

- 자격증 식별자
- 자격증명
- 관련 직무
- 관련 도메인
- 연결 가능한 로드맵 단계
- 추천 요약

### 처리 규칙
- 후보 검색은 canonical recommendation 데이터(JSONL·DB 등 **실행 단계에서 확정**)를 사용한다. **현재는 준비만** — 스키마·예시 파일(`candidates.jsonl.example`)·문서 계약을 갖춘다.
- 추천 결과는 taxonomy 밖 라벨을 포함하지 않아야 한다.
- 추천 결과가 다수일 경우 위험군 단계에 따라 `cert_grade_tier` 기준 정렬을 적용한다: **위험군 단계가 높을수록(4~5단계) 낮은 등급(기능사·산업기사)을 우선 순위로 올리고, 기사·기술사는 후순위로 조정한다.** 위험군 단계가 낮을수록(1~2단계) 기사 이상도 동등 후보로 포함한다.
- `cert_grade_tier`가 null인 자격증은 위험군 단계와 무관하게 정렬 기준 외 후보로 처리한다.
- 일정 정보는 현재 단계에서 기본 출력 필드가 아니다.

### 예외 상황
- 추천 후보 데이터 미존재
- 추천 결과 0건
- canonical store 연결 실패
- 추천 결과에 taxonomy 밖 값 포함

### 현재 상태
활성

## F-04. 추천 이유 / 설명 근거 조회

### 목적
추천 결과에 대해 공식 PDF / HTML 문서 기반 설명 근거를 제공한다.

### 주 사용자
- 사용자

### 입력
- 자격증 식별자
- 선택 입력: 위험군 단계
- 선택 입력: 관련 도메인
- 선택 입력: 관련 직무
- 추천 맥락

### 출력
- 근거 텍스트 snippet
- 관련 문서 식별자
- 선택적 provenance 정보

### 처리 규칙
- 근거 검색은 PDF / HTML 기반 retrieval 계층을 사용한다.
- 구조적 추천 결과와 설명 근거는 분리된 계층에서 생성된다.
- 설명 근거가 없더라도 추천 결과 자체는 반환 가능해야 한다.
- 근거는 추천 이유를 보완하는 용도로 제공한다.

### 예외 상황
- retrieval 결과 0건
- 문서 파싱/인덱싱 누락
- source freshness 문제
- exact match 실패

### 현재 상태
활성

## F-05. 위험군 맞춤 로드맵 조회

### 목적
추천 결과 또는 위험군/도메인 기준으로 단계형 로드맵을 생성한다.

### 주 사용자
- 사용자

### 입력
- 위험군 단계
- 선택 입력: 자격증 식별자
- 선택 입력: 관련 도메인
- 선택 입력: 관련 직무

### 출력
- 로드맵 단계 리스트
- 단계별 설명
- 단계별 연결 자격증 또는 도메인 힌트

### 처리 규칙
- 로드맵은 위험군 단계 기준과 canonical relation 기준을 결합해 생성한다.
- **로드맵은 flat list가 아니라 `cert_prerequisite` 관계(§6.8)를 방향 그래프(DAG)로 순회하여 생성한다.** 사용자 현재 위치(보유 자격증 또는 위험군 단계)에서 실제 이동 가능한 경로만 단계 후보로 포함한다.
- DAG 순회 시 사이클이 존재하는 경우 해당 edge는 무효로 처리하고 순회를 중단한다.
- 현재 단계에서는 일정/링크 없이도 로드맵 결과가 성립해야 한다.
- 로드맵 단계명은 roadmap master 기준을 따른다.

### 예외 상황
- roadmap stage 데이터 누락
- 위험군과 로드맵 연결 관계 없음
- 도메인/직무 기준 불충분

### 현재 상태
활성

## F-08. 시험 일정 조회 (Q-Net API)

### 목적
추천된 자격증과 연결된 시험 일정을 Q-Net 공공 API로 실시간 조회한다.

### 주 사용자
- 사용자

### 입력
- 자격증 식별자 (`cert_id`)

### 출력
- 시험 일정 목록 (회차, 필기시험 기간, 실기시험 기간, D-Day 포함)

### 처리 규칙
- `cert_id` → `cert_master.csv`에서 `cert_name` 조회 → Q-Net `getExamScheduleList` API 호출
- D-Day 계산: `YYYYMMDD` 형식 날짜 → 오늘로부터 남은 일수
- API 키: `hrdkorea_api_key_in` (URL-encoded)

## F-11. 관련 동영상 추천 (YouTube)

### 목적
추천된 자격증의 학습 자료 접근성을 높이기 위해, 자격증명 기반으로 YouTube 강의·인강·합격 영상을 추천한다.

### 주 사용자
- 사용자

### 입력
- 자격증 식별자 (`cert_id`)

### 출력
- 영상 리스트 (최대 5개). 각 영상은 아래 정보를 포함한다.
  - `video_id`: YouTube 영상 ID
  - `title`: 영상 제목
  - `channel`: 채널명
  - `thumbnail_url`: 썸네일 URL
  - `url`: 시청 링크 (`https://www.youtube.com/watch?v={video_id}`)
- 캐시 메타: `fetched_at`, `cache_hit`

### 처리 규칙
- 검색 쿼리는 `cert_name`을 사용해 다음 OR 조합으로 구성한다: `"{cert_name} 강의" | "{cert_name} 인강" | "{cert_name} 합격"`.
- YouTube Data API v3의 `search.list` endpoint를 사용하며 `part=snippet`, `type=video`, `maxResults=5`, `regionCode=KR`, `relevanceLanguage=ko`로 호출한다.
- 검색 결과는 Supabase `cert_video_cache` 테이블(§DATA_SCHEMA.md §6.12)에 캐싱한다.
- TTL: **30일**. `fetched_at`이 30일 이내면 캐시를 반환하고, 만료되면 재호출 후 upsert한다.
- 캐시 키는 `cert_id` 단일. 검색 쿼리 구성 정책이 바뀌면 `query_version` 필드를 증가시켜 무효화한다.
- YouTube API quota(검색 100 units/호출, 일일 10,000 units 무료) 초과 시 캐시된 결과가 있으면 반환하고, 없으면 빈 결과 + `quota_exceeded` 에러 메시지를 응답한다.
- 추천 핵심 흐름과 분리된 부가 기능이므로, 본 기능이 실패해도 추천 자체는 정상 동작해야 한다.

### 예외 상황
- `cert_id`가 candidates에 없음 → 404
- YouTube API 키 미설정 → 500 (운영자 알림 필요)
- YouTube API 쿼터 초과 → 캐시 fallback (없으면 빈 배열 + `quota_exceeded`)
- 검색 결과 0건 → 빈 배열 반환 (오류 아님)
- 네트워크 timeout → 캐시 fallback (없으면 503)

## F-17. 취업지원 자원 번들 조회

### 목적
위험군 단계, 관심 도메인, 관심 직무를 기반으로 취업지원 자원(채용정보·훈련과정·일자리카페·과정평가형)을 번들로 조회하여 Step 3 로드맵 화면에 자격증 추천과 함께 제공한다. 자격증과 시각적으로 구분된 색 테마로 표시한다.

### 주 사용자
- 사용자

### 입력
- 위험군 단계 ID (`risk_stage_id`, 예: `risk_0004`) — 필수
- 관심 도메인 ID (`domain_id`) — 선택
- 관심 직무 ID 목록 (`job_ids`) — 선택
- 지역명 (`region`, 예: `서울`) — 선택 (없으면 전국)
- 연결 자격증 ID 목록 (`cert_ids`) — 선택 (훈련과정 NCS 파생에 활용)

### 출력
- `resource_types`: 이번 요청에서 활성화된 자원 유형 목록
- `support_level`: 지원 수준 레이블 (`partial` / `standard` / `full`)
- `bundles`: 자원 유형별 번들 배열
  - `resource_type`: `hiring` | `training` | `job_cafe` | `process_eval`
  - `label`: 화면 표시 레이블 (예: "채용정보", "훈련과정", "일자리카페", "과정평가형 자격")
  - `items`: 실제 조회 결과 (각 서비스 응답 구조와 동일)
  - `count`: 항목 수

### 처리 규칙 — 위험군별 자원 유형 게이팅

| 위험군 단계 | 지원 수준 | 활성 자원 유형 |
|---|---|---|
| 1단계 (취업 안정권) | `partial` — 취업 정보 일부 | `hiring` |
| 2~3단계 (중간 위험군) | `standard` — 역량 강화 지원 | `hiring`, `training` |
| 4~5단계 (고위험군) | `full` — 다지원 | `hiring`, `training`, `job_cafe`, `process_eval` |

- `hiring` 조회: 관심 직무 이름을 keyword로 WorkNet API 호출. 직무 미지정 시 도메인명을 keyword로 사용.
- `training` 조회: 도메인 → NCS 1차 분류 매핑 테이블로 `ncs_category` 파생 후 Work24 API 호출.
- `job_cafe` 조회: 서울시 열린데이터광장 API. 지역 미지정 시 전체 조회.
- `process_eval` 조회: Q-Net 과정평가형 자격 목록. 도메인/직무 필터 없음.
- 각 자원 조회가 실패(API 키 미설정·타임아웃)해도 다른 자원 조회는 계속 진행하며, 실패한 자원은 `items: []` + `error` 필드로 반환한다.
- 이 기능의 실패가 로드맵 자격증 추천 자체의 실패를 유발하지 않는다.

### 도메인 → NCS 1차 분류 매핑 (대표 예시)

| 도메인 taxonomy | NCS 1차 분류 |
|---|---|
| 데이터/AI | 정보통신 |
| 소프트웨어개발 | 정보통신 |
| 금융/핀테크 | 금융/보험 |
| 건설/토목 | 건설 |
| 기계/제조 | 기계 |
| 화학/바이오 | 화학/바이오 |
| 전기/전자 | 전기/전자 |
| 경영/사무 | 경영/회계/사무 |
| 보건/의료 | 보건/의료 |

매핑 불일치 시 `ncs_category = null`로 Work24 전체 조회.

### 예외 상황
- `risk_stage_id` 미제공 → `MISSING_REQUIRED_FIELD`
- 개별 외부 API 실패 → 해당 번들 `items: []`, `error` 메시지 포함, 나머지 정상 반환
- 모든 번들 `items: []` → 오류 아님, 빈 번들 응답 정상 반환

### 현재 상태
활성

## 7. 공통 예외처리 원칙

모든 사용자 기능은 아래 공통 원칙을 따른다.

1. 추천 결과가 0건이어도 시스템 오류와 구분해야 한다.
2. 문서 근거가 없더라도 추천 결과는 반환 가능해야 한다.
3. taxonomy 밖 값이 생성되면 validation error로 간주한다.
4. reserved 기능은 명시적으로 비활성 상태로 처리한다.
5. 현재 스프린트 범위 밖 기능을 임의로 fallback 구현하지 않는다.

## 10. 최종 요약

이 문서는 시스템이 제공하는 기능을 **사용자 기능 단위**로 정의한다.  
핵심 추천 흐름(F-01~F-07, F-11)과 Execution Layer(F-08~F-09, F-12~F-16)가 모두 활성화되었다.

현재 시스템의 기능 중심은 아래 세 축이다.

1. **추천 가능한 구조적 데이터 생성** (F-06~F-07)
2. **추천 결과를 사용자 흐름으로 제공** (F-01~F-05, F-11)
3. **추천 결과에서 실행 행동으로 이어지는 Execution Layer** (F-08~F-09, F-12~F-16)

## 11. 데모 제출용 기능 단계 (임시)

> 본 절은 `PRD.md` **§19** 와 대응한다. **제출·시연 기한 동안만** 구현 우선순위로 사용한다.  
> 기능 ID(F-xx) 정의는 상기 본문이 우선이며, 본 절은 **최소 동작·허용 스텁**만 추가한다.

### 11.1 단계·기능 매핑

| 단계 | 목표 | 관련 기능 | 데모 최소 완료 기준 |
|:---:|:---|:---|:---|
| D1 | 시나리오·데이터 고정 | F-01, F-02 | 시연에 쓸 입력값·taxonomy 조합·문서 목록이 고정됨 |
| D2 | 인덱싱 재현 | F-04(검색 입력 전제) | Parse→Chunk→Embed→Store가 문서화된 명령으로 재실행 가능 |
| D3 | 근거 검색 경로 | F-04 | `cert_id`(또는 동등 키)로 snippet 1건 이상 조회 가능; 0건은 예외로 처리 |
| D4 | 추천·로드맵 시연 | F-03, F-05 | 후보 ≥1, 로드맵 단계 ≥1 표시 |
| D5 | LLM 조립(임시) | F-04 | 검색·canonical 필드만으로 응답 생성; 근거 없을 때 사실 생성 금지 (`PROMPT_DESIGN.md`) |
| D6 | 제출·재현 패키지 | F-06~F-07(선택) | subset canonical만으로도 D4가 성립하면 전체 배치는 생략 가능 |

### 11.2 데모 기간 허용 스텁

- **F-06~F-07**: 전체 CSV가 아닌 **데모용 subset 파일**만 canonical/candidate에 반영해도 된다. 단, taxonomy 검증 규칙은 완화하지 않는다.  
- **F-10**: REST/화면 대신 **파일 산출물·CLI exit code**로 validation을 보여도 된다.  
- **F-08~F-09**: Q-Net API 연동 완료 (2026-05-14). 데모에서 실제 일정 데이터 표시 가능.

### 11.3 데모에서 금지하는 것

- reserved 기능을 **완료된 것처럼** 노출하거나 실데이터 일정·링크를 지어내기.  
- retrieval 0건인데 LLM이 **문서에 없는 규정·날짜·URL**을 채우기.  
- taxonomy 밖 `related_jobs` / `related_domains`를 사용자 응답에 포함하기.

- 본 절(§11)을 삭제하거나, MVP 본문(§5~§8)과 합치면서 중복 표를 정리한다.
